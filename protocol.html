Archive Protocol

This page documents the operational protocol governing the Signal Archive. The protocol is stable and has not been modified since the initial commit.

Generation

Before each market window, the model produces one or more signals. Each signal predicts the outcome of a specific match under a specific handicap. At signal generation, the corresponding Crown odds (皇冠赔率) are captured. The batch is serialized as a CSV, compressed and encrypted into a zip file, and committed to this repository.

The time window between signal generation and match kickoff is always at least thirty minutes. The commit timestamp of each encrypted signal file, as recorded by GitHub, is the canonical proof of that ordering.

Settlement

Match results are recorded in track_record.csv within twenty-four hours of match conclusion. At the same commit, the password for the corresponding zip file is published in track_record.csv, making the original encrypted signal fully verifiable against the row that references it.

Structural Invariants

The archive is governed by five invariants. Violation of any invariant invalidates the archive.

  1. Sequential Signal IDs. Every signal is assigned a strictly monotonically increasing integer ID. No gaps, no reuse, no reordering.
  2. Count Equality. The number of commits to the /csv directory equals the number of zip files equals the number of rows in track_record.csv.
  3. Monthly Snapshots. At the end of each calendar month, a snapshot commit is made with cumulative counts and metrics. Snapshots are immutable once committed.
  4. Odds Verifiability. Crown odds recorded at signal time are independently cross-verifiable against third-party odds history databases.
  5. Coverage Commitment. Signal scope is defined by objective model criteria fixed in advance, not by post-hoc selection. The criteria are documented in the repository and cannot be silently narrowed.

Verification

For verifying a specific signal:

  1. Locate the signal's row in track_record.csv. Note the zip filename and password.
  2. Download the zip from the /csv directory in this repository.
  3. Check the commit timestamp of the zip — it must precede match kickoff.
  4. Extract the zip using the password. Compute SHA256 of the inner CSV.
  5. Compare the content to the corresponding row in track_record.csv.

A convenience script verify.py is provided in the repository root for batch verification.

Statistical Considerations

The archive is a single public record, committed to this repository from the beginning and cross-referenced externally via domain registration, account history, and commit chronology. Under the hypothesis that the archive is one selected from many parallel private archives, the number of parallel archives required to yield the observed ROI grows super-exponentially with sample size and ROI magnitude.

At N ≥ 1000 signals and sustained ROI ≥ 10%, the number of candidate parallel archives required to explain the record by selection alone exceeds any plausible budget of parallel experimentation. The relevant figures will be restated, with the precise multi-testing correction, on each monthly snapshot.
