档案协议

本页记录信号档案的运行协议。协议自首次提交以来保持稳定，未作修改。

生成

每次市场窗口开始前，模型产出一条或多条信号。每条信号针对特定赛事在特定盘口下的结果进行预测。信号生成时，同步记录该赛事的皇冠赔率。整批信号序列化为 CSV 文件，压缩加密后以 zip 格式提交至本仓库。

信号生成至赛事开球之间的时间间隔，不少于三十分钟。每个加密文件的 GitHub commit 时间戳，即为这一时序的规范证明。

结算

赛果在比赛结束后的 24 小时内录入 track_record.csv。同一次 commit 中，对应 zip 文件的解密密码一并公开，使原始信号可被任何人独立验证。

结构不变量

档案遵循以下五条不变量，任一被违反即视为档案作废。

  1. 信号序号连续。每条信号分配严格递增的整数序号，不可跳号、不可复用、不可重排。
  2. 数量一致。/csv 目录下的 commit 数 = zip 文件数 = track_record.csv 行数。
  3. 周期快照。每月末提交一次累计快照 commit，包含截至当时的全部计数与指标。快照一经提交即不再修改。
  4. 赔率可核验。信号生成时刻所记录的皇冠赔率，可在第三方赔率历史数据库中独立核验。
  5. 覆盖承诺。信号范围由客观的模型规则预先定义，不做事后挑选。规则记录在仓库中，不可在不留痕的情况下收窄。

验证

验证任一条信号的步骤如下。

  1. 在 track_record.csv 中定位该信号所在行，记录其对应的 zip 文件名与密码。
  2. 从仓库的 /csv 目录下载该 zip 文件。
  3. 查看该 zip 文件的 commit 时间戳——应早于该场赛事开球时间。
  4. 使用密码解压，计算内部 CSV 文件的 SHA256。
  5. 将其内容与 track_record.csv 中对应行比对。

仓库根目录提供脚本 verify.py，用于批量执行上述步骤。

统计学考量

本档案是一份单一的公开记录，自项目开始即提交至本仓库，并通过域名注册信息、账号创建时间、commit 时序等外部路径交叉锚定。

若假设本档案是从大量未公开的平行档案中事后挑选出来的，产出同等表现所需的平行档案数量，随样本量与 ROI 量级呈超指数增长。当 N ≥ 1000、ROI 稳定在 10% 以上时，所需平行档案数已超出任何个人或组织可能并行运营的规模。具体的多重检验修正值，将在每月快照中更新披露。
